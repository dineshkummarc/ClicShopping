text_sql_generation_rules = Lors de la génération d’une requête SQL :
1. Toujours utiliser les préfixes complets des tables (par exemple, `clic_products` au lieu de `products`).
2. Ajouter les jointures appropriées pour les tables liées.
3. Filtrer par `language_id` lorsque pertinent.
4. Optimiser la requête pour de bonnes performances.
5. Ajouter les clauses `ORDER BY` appropriées.
6. Limiter les résultats à un nombre raisonnable si nécessaire (`LIMIT`).
7. Si un champ texte est impliqué dans une condition (comme un nom ou une description), utiliser l’opérateur `LIKE` avec des caractères génériques (%) pour permettre des recherches partielles.
8. IMPORTANT POUR LES NOMS DE PRODUITS : Lors de la recherche de produits par nom, UTILISER TOUJOURS `LIKE` avec des jokers :
   - Nom unique : `WHERE pd.products_name LIKE '%ProductName%'`
   - Plusieurs mots : utiliser `AND` pour une correspondance exacte : `WHERE pd.products_name LIKE '%Word1%' AND pd.products_name LIKE '%Word2%'`
   - Orthographe alternative : utiliser `OR` pour les variations : `WHERE pd.products_name LIKE '%Josef%' OR pd.products_name LIKE '%Joseph%'`
   - Exemples :
     - "price of product Picardie" → `WHERE pd.products_name LIKE '%Picardie%'`
     - "Josef Strauss product" → `WHERE pd.products_name LIKE '%Josef%' AND pd.products_name LIKE '%Strauss%'`
     - "Joseph Strauss" → `WHERE (pd.products_name LIKE '%Josef%' OR pd.products_name LIKE '%Joseph%') AND pd.products_name LIKE '%Strauss%'`
9. IMPORTANT POUR ÉVITER L'AMBIGUÏTÉ : Toujours préfixer les colonnes avec l'alias de la table dans les JOINs :
   - Utiliser `p.products_id` au lieu de `products_id`
   - Utiliser `p.products_price` au lieu de `products_price`
   - Utiliser `pd.products_name` au lieu de `products_name`
   - Exemple correct : `SELECT p.products_price AS catalog_price, p.products_id FROM clic_products p JOIN clic_products_description pd ON p.products_id = pd.products_id`
10. S’assurer que la requête est correcte avant exécution et qu’il n’y a pas d’injections SQL.
11. Si vous détectez des incohérences (doublons, sommes incorrectes, données manquantes), alerter l’utilisateur avec une recommandation spécifique sur la validation des données ou l’amélioration des requêtes SQL.
12. IMPORTANT POUR LES COMPARAISONS TEMPORELLES : Lors de la comparaison de métriques entre différentes périodes (par exemple, "mai vs février", "T1 vs T2", "janvier vs mars") :
    - Utiliser `CASE WHEN` avec `SUM()` pour créer des colonnes séparées pour chaque période
    - Exemple : `SUM(CASE WHEN MONTH(date) = 5 THEN value ELSE 0 END) AS may_revenue`
    - Toujours inclure un filtre `YEAR` pour éviter de mélanger des données d'années différentes
    - Retourner les résultats sur une seule ligne avec plusieurs colonnes (une par période)
    - NE PAS créer des requêtes séparées pour chaque période
