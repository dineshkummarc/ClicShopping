Requête : {{QUERY}}

Considérez ces types d'ambiguïté :
    1. Quantification : COUNT (nombre d'éléments distincts) vs SUM (quantité totale/valeur).
    2. Portée (Scope) : Tous les éléments vs Éléments récents vs Éléments actifs/visibles.
    3. Métrique : Moyenne vs Minimum vs Maximum vs Liste simple.
    4. Période temporelle : Plage de temps non spécifiée.

  "ambiguity_type": "quantification|scope|metric|time|null",
  "confidence": 0.0-1.0,
  "interpretations": [
    {
      "type": "count|sum|all|recent|active|average|min|max|list",
      "label": "Short label",

  "reasoning": "Brief explanation"
}

Règles :
    - Si la requête mentionne explicitement COUNT, DISTINCT, SUM, TOTAL, AVG, etc. -> NON ambiguë.
    - Si la requête spécifie la période temporelle, la portée ou la métrique -> NON ambiguë.
    - Si la requête est vague sur ce qu'il faut compter/sommer/moyenne -> EST ambiguë.
    - Fournir 2 à 4 interprétations si ambiguë.
    - Recommander "generer_les_deux" pour l'ambiguïté de quantification (COUNT vs SUM).
    - Recommander "utiliser_defaut" pour l'ambiguïté de portée/métrique, en privilégiant l'interprétation la plus courante.
    - Recommander "clarifier" pour l'ambiguïté temporelle ou si la requête est trop vague pour une interprétation par défaut.

Exemples :
    - "Combien de produits en stock ?" -> AMBIGUË (compter les modèles vs sommer la quantité).
    - "Compter les produits distincts" -> NON ambiguë.
    - "Afficher les commandes" -> AMBIGUË (toutes vs récentes vs actives).
    - "Afficher toutes les commandes" -> NON ambiguë.
